# AI Sustainable Commerce Backend System

## Overview

This project implements AI-powered backend modules designed to automate product catalog classification and generate sustainability impact insights.

The goal of the system is to demonstrate how **AI can be integrated into a real backend architecture** to reduce manual catalog management, automate sustainability reporting, and improve business workflows.

The system is designed using a **clean layered architecture** where AI logic, business logic, and API handling are separated.

---

# Implemented Modules

The assignment required implementing **any two modules**.
The following modules have been fully implemented.

---

# Module 1 – AI Auto Category & Tag Generator

This module automatically classifies products using an AI model and generates structured metadata.

### Features

* Detects product category
* Suggests sub-category
* Generates SEO tags
* Suggests sustainability filters
* Returns structured JSON output
* Stores result in MongoDB
* Logs AI prompt and response

### Example Request

POST `/api/generate-category`

```json
{
"name":"Bamboo Toothbrush",
"description":"Eco friendly bamboo toothbrush"
}
```

### Example Response

```json
{
"name":"Bamboo Toothbrush",
"description":"Eco friendly bamboo toothbrush",
"category":"Personal Care",
"subcategory":"Oral Care",
"tags":["eco-friendly","bamboo","plastic-free"],
"sustainability_filters":["biodegradable"]
}
```

# Module 3 – Sustainability Impact Reporting Generator

This module estimates environmental impact generated by product usage.

### Features

* Calculates plastic waste reduction
* Estimates carbon emissions avoided
* Generates readable sustainability statement
* Stores result in MongoDB

### Example Request

POST `/api/impact-report`

```json
{
"product":"Bamboo Bottle",
"quantity":10
}
```

### Example Response

```json
{
"product":"Bamboo Bottle",
"quantity":10,
"plastic_saved":1000,
"carbon_avoided":15,
"impact_statement":"This order avoided 1000 plastic bottles"
}
```

# Architecture

The system follows a **clean architecture pattern** separating API, AI services, and database models.

```
Client / API Consumer
        │
        ▼
Express API Routes
        │
        ▼
Controllers
        │
        ▼
Services Layer
(AI Logic + Business Logic)
        │
        ▼
Database Models
        │
        ▼
MongoDB
```

### Key Design Principle

AI logic is isolated inside the **services layer** so that:

* controllers remain lightweight
* AI models can be replaced easily
* business logic stays independent of API logic

# Environment Variables

Create a `.env` file in the root directory.

```
PORT=5000
MONGO_URI=your_mongodb_connection_string
GROQ_API_KEY=your_groq_api_key
```

API keys are stored using environment variables to ensure security and avoid exposing secrets in the repository.

---

# AI Prompt Design

The AI prompt enforces **strict JSON output** to ensure reliable parsing and database storage.

Prompt rules include:

* Return only JSON
* No explanations
* No markdown
* Follow predefined structure

This ensures that the AI response can be programmatically parsed and stored.

---

# Prompt & Response Logging

To monitor AI behaviour and improve reliability, each request logs:

* Prompt sent to AI
* Response returned by AI
* Timestamp

These logs are stored in the **ai_logs collection**.

Example document:

```
{
prompt: "...",
response: "...",
created_at: "2026-03-05"
}
```

---

# Structured JSON Output

All AI responses are validated and parsed into structured JSON before being stored in the database.

This ensures compatibility with backend systems and prevents malformed responses.
